FYI — Session type differences: server-side SDK vs CLI thread vs client wrapper vs stateless API
Server-side SDK
Claude · Agent SDK
The provider's server owns the conversation state. You call the SDK, pass your message, and the server appends it to a persistent thread. On the next turn, you pass only the new message — not the full history. The server already has it.

The thread has an ID you can resume across process restarts. Context is never lost unless you explicitly clear it or the session expires.
+ No history management on your side
+ Resumable across restarts (thread ID)
+ No growing payload per turn
− Proprietary SDK, not portable
− Server-side session can expire
− Harder to inspect what the model actually received
CLI thread resume
OpenAI · Codex CLI
Similar to server-side SDK but accessed via a CLI subprocess rather than a library. The CLI maintains a thread ID on disk. You resume a session by passing --thread-id to the next invocation. The server holds the history.

Kollab drives this via subprocess. The thread ID is stored per session and reused on each turn. No history replay needed.
+ Server holds history, no payload growth
+ Resumable via thread ID
+ Works without an SDK dependency
− Subprocess overhead per turn
− No MCP support in exec mode
− Less control than direct SDK
Client-side wrapper
Google · Gemini ChatSession
A convenience object in the Google GenAI SDK that accumulates history[] for you locally. Under the hood, every API call sends the full accumulated history to a stateless endpoint. The server has no memory between calls.

Looks like a session from the outside, but it's just history management in the SDK layer. If your process restarts, the history is gone.
+ Convenient SDK abstraction
+ No manual history tracking needed
− History lost on process restart
− Payload grows every turn (cost + latency)
− Not truly resumable — no server-side thread
− Treated as stateless in kollab
Stateless API
DeepSeek · Grok · Groq · Mistral · Perplexity · others
Pure stateless REST. Each call is independent. You send a messages[] array containing the full conversation history yourself. The server processes it and responds. It remembers nothing between calls.

Multi-turn behaviour is achieved by the caller accumulating and replaying history on every request. This is what all these providers' chat UIs do behind the scenes — but it's your responsibility when calling the API directly.
+ Simple, portable, no SDK required
+ Full visibility into exactly what was sent
+ Easy to switch providers (same messages[] shape)
− You own history management
− Payload and cost grow with conversation length
− Not suitable for kollab's multi-round adversarial loop
− Fine for single-stage task roles only
kollab rule: Server-side SDK (Claude) and CLI thread resume (OpenAI) are the only session types eligible for multi-round Producer/Critic roles. Client-side wrapper (Gemini) and all stateless providers are task-role only — context is injected once per turn, one turn per stage.
